Decisions/patterns:
- D-17: each active robot = one asyncio.create_task in the FastAPI process; no Celery/Redis/subprocess.
- D-01/D-04: ONE FastAPI WS per user at /ws/{user_id}, messages multiplexed by robot_id envelope.
- D-02/D-03: persisted order/status events via Supabase Realtime; equity updates per order event.
- D-05/D-06: Simulado uses live ticks from the shared system MetaAPI account; no user broker link needed.
- RESEARCH Pattern 2 (WS manager), Pattern 4 (RobotEngine state machine).
- RISK-02: on reconnect → get_positions/get_orders, RESET all strategy in-memory state; staleness sentinel
  (no tick >10s on active symbol → halt order placement); deterministic clientOrderId.
- RISK-03 / Pitfall 1: ring buffer collections.deque(maxlen=3600) per symbol; raw ticks NEVER written to Supabase;
  OHLCV flush every 1s only. Pitfall 3: all in-memory state keyed connections[user_id][robot_id].
- Pitfall 6: zoneinfo America/Sao_Paulo for all trading-hour comparisons.
- EXE-04 contract resolution (RESEARCH Integration Touchpoint 4): query b3_contracts is_front_month=true;
  APScheduler daily 07:00 BRT updates front-month + triggers rollover; force-close expiring by 16:50 BRT on expiry day.
- PRD §18 RF-EXE-05: fill policies Pessimista (default) / Moderado / Otimista — read §18 for fill mechanics; do not re-derive.
- idempotency: client_order_id = sha256(user_id + robot_id + signal_timestamp); DB UNIQUE rejects duplicates.
  State machine QUEUED→SENDING→SENT→CONFIRMED→FILLED.
