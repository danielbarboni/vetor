<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Supabase REST/Realtime | Authenticated client reads/writes user data; RLS is the enforcement point |
| browser → FastAPI REST/WS | Bearer JWT crosses here; validated server-side (see plan 03) |
| FastAPI → MetaAPI | Broker credentials never leave the backend; only MetaAPI account ID stored (plan 09/10/13) |
| FastAPI → Supabase (service role) | Backend writes orders/status; bypasses RLS — must always scope by user_id in code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Info Disclosure | orders/robots tables (user A reads user B) | mitigate | RLS USING (user_id = auth.uid()) on ALL user-scoped tables; smoke-tested in Task 3 checkpoint |
| T-01-02 | Tampering | orders duplicate injection on retry | mitigate | orders.client_order_id UNIQUE constraint (DB-level idempotency); enforced again at adapter (plan 10) |
| T-01-03 | Spoofing | JWT forgery on API/WS | mitigate | Supabase JWT validated against SUPABASE_JWT_SECRET, audience="authenticated" (plan 03) |
| T-01-04 | Elevation | service-role backend writes bypass RLS | mitigate | backend always filters by user_id from validated JWT; never trusts client-supplied user_id (plans 05/10) |
| T-01-05 | Tampering | SQL injection via JSONB params | mitigate | parameterized supabase-py queries; Pydantic validates JSONB before write (plans 05/08) |
| T-01-06 | Info Disclosure | b3_contracts writable by client | mitigate | b3_contracts SELECT-only policy for authenticated; no client insert/update |
| T-01-SC | Tampering | npm/pip installs | mitigate | package legitimacy checkpoints for [ASSUMED] python-jose, APScheduler (plan 03); echarts pinned 6.1.0 (plan 01) |
</threat_model>
